What is BitDiver?

BitDiver is an open-source test automation framework for declarative, data-driven End-to-End testing. It consolidates configuration management, suite building, step execution, result checking, and logging into a single TypeScript package. Define test suites in YAML or JSON, register custom step implementations, and run them with configurable parallelism. BitDiver supports three step types — normal (per testcase), single (per suite), and Runner-managed timing — with built-in persistence, Zod-validated configuration, and pluggable log adapters. Used by enterprises including Deutsche Bahn for automated regression testing.

WORKFLOW /// HOW IT WORKS
01

Define Your Suite

Create a YAML or JSON configuration that declares your test steps, data mappings, and execution order. Use timed file scanning to automatically generate time-based steps from filename prefixes.

02

Register Your Steps

Implement custom step classes extending StepNormal or StepSingle. Register them in the StepRegistry with a name that matches your suite configuration. Built-in timing steps are pre-registered.

03

Run Your Tests

Execute the suite with the Runner. It handles parallel execution, progress reporting, and result collection. Check results automatically with deep comparison and get detailed summaries.

N

StepNormal

Executes once per testcase. Access testcase context via typed tc getter. Ideal for data-driven operations like sending requests or validating responses.

TYPE / PER-TC
S

StepSingle

Executes once for the entire suite. Access all testcases via typed testcases getter. Use for setup/teardown like database initialization or environment preparation.

TYPE / PER-SUITE
T

Runner-Managed Timing

The Runner controls timing via built-in steps: DetermineStartTime calculates when to start, CheckStartTime verifies the budget, and Wait pauses execution. Steps carry an offsetSeconds timing property.

EXECUTION /// MODES
B

Batch Mode (Default)

Column-based execution: iterate steps, then testcases per step. Optimized for parallel execution — all testcases run the same step concurrently before moving to the next step.

MODE / COLUMN
R

Normal Mode

Row-based execution: iterate testcases, then steps per testcase. Each testcase runs all its steps sequentially before the next testcase starts.

FAQ /// COMMON QUESTIONS
What is BitDiver?

BitDiver is an open-source test automation framework for declarative, data-driven End-to-End testing. It consolidates configuration, suite building, step execution, result checking, and logging into a single TypeScript package. Define suites in YAML/JSON, register custom steps, and run with parallel execution.

How do Steps work?

Steps follow a lifecycle: start() → beforeRun() → run() → afterRun() → end(). There are two primary types: StepNormal (runs per testcase) and StepSingle (runs once per suite). Timing is handled by the Runner using built-in steps (DetermineStartTime, CheckStartTime, Wait). Custom steps extend these base classes and implement the run() method.

What are the execution modes?

BitDiver supports two modes. Batch mode (default) iterates steps first, then testcases per step — optimized for parallel execution. Normal mode iterates testcases first, then steps per testcase — each testcase completes all steps before the next starts.

How do I write a custom Step?

Create a TypeScript class extending StepNormal or StepSingle. Implement the run() method with your test logic. Register the step in the StepRegistry with a name matching your suite configuration. Access testcase data via the typed tc getter (StepNormal) or testcases getter (StepSingle). Built-in timing steps are pre-registered and available by default.

How does configuration work?

BitDiver uses Zod schemas for configuration validation with TypeScript type inference. The loadConfig() function supports a priority system: environment variables override inline values, which override file values, which override schema defaults. Sensitive fields are automatically masked in logs.
